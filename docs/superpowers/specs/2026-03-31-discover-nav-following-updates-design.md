# 发现（左侧抽屉）入口 + 关注更新提示 - 设计稿

日期：2026-03-31  
范围：`J:\code\tuan-chat-sandbox\material-package-nav\tuan-chat-web-main`

## 背景与目标

当前“发现”页左侧抽屉（`ChatDiscoverNavPanel`）仅包含「广场 / 我的归档」两个入口。希望在这里增加两个“发现入口”：

- **素材包广场**
- **仓库（模组）广场**

并在入口上提供一个**不打扰**的“关注更新提示”，用于告诉用户：关注的人在对应广场是否有新内容，并允许用户进一步展开查看更新条目。

### 设计目标

- 默认视觉简洁：抽屉不变成动态流。
- 一行入口承载两种意图：
  - 点击行主体：进入对应广场列表页（“去逛”）。
  - 点击右侧提示按钮：展开“关注更新条目”（“看更新”）。
- 移动端友好：统一使用**单击**触发，不使用双击。

### 非目标（本次不做）

- 不引入“我的管理”页面（已决定删除）。
- 不强制实现完整的“查看全部更新”页面（可后续扩展）。
- 不处理复杂的“按人聚合”视图（先用按条目时间倒序）。

## UI 结构（选型：B）

`ChatDiscoverNavPanel` 结构调整为两块：

1. **发现入口**（一张卡片，内部两行）
   - 素材包广场（行）
   - 仓库广场（行）
2. **归档仓库**（沿用现有两项）
   - 广场
   - 我的归档

每个“发现入口行”右侧提供一个**提示按钮（胶囊 + ▾）**，作为明确可点击热区。

## 交互细节

### 热区与可访问性规范（实现约束）

- 每个“发现入口行”拆成两个可聚焦控件：
  - **主热区按钮**：覆盖标题/副标题区域以及行内空白（不包含右侧提示按钮）。
  - **提示按钮（胶囊 + ▾）**：独立点击区，仅负责展开/收起更新列表。
- **最小触摸/点击尺寸**：
  - 主热区行高建议 ≥ `44px`
  - 提示按钮建议 ≥ `32px` 高（移动端建议 ≥ `36px`），并留出左右 padding 便于点击
- **键盘可达**：
  - Tab 顺序：先主热区按钮，再提示按钮
  - Enter/Space：
    - 在主热区按钮上：触发跳转到对应广场列表页
    - 在提示按钮上：触发展开/收起更新列表
  - 为两个按钮分别设置清晰的 `aria-label`（例如“进入素材包广场”“展开素材包广场关注更新”）
- **点击归属**：
  - 行内除提示按钮外的任何区域（包含右侧空白）都归属主热区，避免“点空白无响应”的割裂感。

### 入口行

- **点击行主体（不含提示按钮）**：进入对应广场列表页。
  - 素材包广场：跳转 `/chat/material-package`
  - 仓库广场：跳转 `/chat/discover`

### 右侧提示按钮（关注更新）

- 单击提示按钮：展开/收起该入口行下方的“关注更新列表”。
- 展开后：在该行第二行位置下方渲染一个列表容器。
- 列表展示：
  - **按更新条目时间倒序**
  - **最多显示 5 条**
  - 每条包含：
    - 更新者头像
    - 一句话摘要（谁更新了什么）
    - 简短说明（可选，尽量短）
    - 时间（相对或本地化时间均可）
- 点击某条更新：直达对应详情（优先打开既有右侧详情抽屉；否则退化为跳转到详情页/列表定位）。

#### 展开/滚动策略（移动端优先）

- **互斥展开（默认）**：同一时间只允许一个入口行展开更新列表；展开另一个时自动收起前一个。
- 列表不做内嵌滚动：最多 5 条，随抽屉整体滚动，避免“抽屉滚动 vs 列表滚动”冲突。

### 视觉收敛策略

- 默认态（未展开）：入口行保持简洁。
- “有更新”时才出现提示按钮内的提示（红点/数量/头像预览）。
- “无更新”时**仍保留提示按钮占位**（弱态），点击展开显示空态“暂无更新”，避免布局抖动。

> 注：如果默认态头像堆叠导致视觉混乱，可降级为“红点 + 数量（● 3）”，头像只在展开列表中出现。

## 数据与状态

### 更新条目数据模型（前端视角）

统一抽象为：

- `type`: `"materialPackage" | "repository"`
- `actorUserId` / `actorAvatarUrl`
- `targetId`（`packageId` 或 `repositoryId`）
- `title`（目标名称）
- `summary`（一句话摘要）
- `timestamp`

### 数据来源（分阶段）

#### 阶段 1（先把交互跑通）

- **仓库广场更新**：复用现有“关注动态 feed”接口 `POST /feed/moment`（前端已有 `useGetFollowingMomentFeedInfiniteQuery`）。
  - 仅拉取首页：`pageSize` 取 `20`，读取返回 `list` 的前若干条用于统计/展示。
  - 条目过滤（最小可用）：`FeedWithStatsResponse.type` 在仓库相关集合内，且 `response.repositoryId` 为有效 ID。
    - 初版推荐集合：`{ 5, 6, 7, 8 }`（仓库发布 / fork / 贡献 / 收藏 等，若实际后端含义不同可收敛为仅 `5`）
  - 字段映射（尽量不额外请求）：
    - `actorUserId`：`response.userId`（若缺失则置空并不展示头像）
    - `targetId`：`response.repositoryId`
    - `title`：`response.name`（仓库名）或回退为 `仓库 #id`
    - `summary`：优先用 `response.description` 或用 `type` 映射出的短句（如“发布了仓库”）
    - `timestamp`：`response.createTime` / `response.updateTime`（择其一）
- **素材包广场更新**：阶段 1 用 mock（本地/测试模式）渲染交互与 UI；待接口补齐再接入。

#### 阶段 2（后端接入）

为素材包广场补齐“关注更新 feed”接口（按关注用户的素材包发布/更新聚合），前端替换 mock 数据源。

##### 接口最小契约（建议）

- 请求：`GET /materialPackage/following/updates?limit=5&cursor=...`
  - `limit`：默认 5
  - `cursor`：可选游标（时间/ID）
- 返回：
  - `items: Array<{ actorUserId; actorAvatarUrl?; packageId; packageName; summary?; timestamp }>`
  - `nextCursor?: string`
- 去重键建议：`packageId + timestamp + actorUserId`（或由后端提供稳定 `eventId`）
- 前端拉取策略：
  - 抽屉渲染时可先取“是否有更新 + 数量”的轻量字段（可选优化）
  - 展开时再拉取详情条目（推荐），并设置 `staleTime`（如 1~5 分钟）

## 错误处理与边界

- 未登录 / userId 缺失：入口仍可用；提示按钮弱态，展开后显示“登录后可查看关注更新”。
- 请求失败：入口仍可用；展开列表显示“加载失败，点此重试”，提供重试按钮（不阻塞跳转）。
- 列表为空：展开容器展示空态文案（例如“暂无更新”），不展示头像。
- 多入口同时展开：默认**互斥展开**（一次只展开一个），避免移动端抽屉被占满。
